## Summary

This is the first committed Goal 48 code-dataset L4 report. It validates the
cached sparse training path on a fixed code split and records Dense, MoP Full,
warm sparse, and cached sparse metadata in a lightweight report folder.

Key measured points:

- Cached Warm Adapter Norm/Head 64 reached `85,669.1560` tokens/sec versus
  Dense at `7,785.0067` tokens/sec.
- Cached Warm Adapter Norm/Head 64 used `0.0586 GB` peak reserved VRAM versus
  Dense at `1.8516 GB`.
- Cached Warm Adapter Norm/Head 64 used a `5.0355 MB` trainable-only checkpoint
  versus Dense at `987.1504 MB`.
- Cached Warm Adapter Norm/Head 64 trained about `0.54%` of parameters.
- Cached training offloaded `93,129,216` frozen backbone parameters.

Important caveats:

- `target_eval_loss` was not configured, so time-to-target-loss is not measured
  in this report.
- Generated-code verifier and exact-match quality did not pass for the compared
  non-cached runs, and cached-tail generation quality is unavailable because the
  cached-tail phase has no source lessons for generation evaluation.
- Cached sparse was dramatically lighter and faster, but this report should be
  treated as evidence that the cached sparse GPU-efficiency path works, not as
  proof of same-quality sparse superiority.

## Interpretation Rules

This report supports only measured claims. A cached sparse run is a same-quality efficiency win only if it remains close to Dense eval loss and generated-code quality while improving a named efficiency axis such as VRAM, trainable parameters, checkpoint size, or time-to-target-loss.

For this run, the strongest claim is narrower:

```text
MoP-Forge can run cached sparse teacher-distilled code training on L4, offload
the frozen backbone during the cached-tail phase, and report large measured
efficiency improvements on VRAM, throughput, trainable parameters, and
checkpoint size. Same-quality generated-code efficiency is not proven here.
```
